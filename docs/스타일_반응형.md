# auto-dr — Design & Responsive Implementation Brief

> **Purpose of this document.** This is the single source of truth for redesigning the auto-dr dashboard so it reads as *human-crafted, distinctive, and unmistakably professional* — not as a generic AI-generated template. It is written to be executed directly against `src/dashboard/public/{index.html,styles.css,app.js}` with no external CDN, no framework, and no build step (the project runs `tsx` + a local Express/SSE server). Treat every section below as a requirement, not a suggestion.

---

## 0. Hard constraints (read first)

These are non-negotiable properties of the project. The design must live inside them.

- **No external network.** The dashboard is served locally and must work fully offline. **No Google Fonts, no CDN CSS/JS, no remote images.** Any custom font must be self-hosted under `public/fonts/` and loaded with `@font-face` + `font-display: swap`, or we fall back to a high-quality system stack (defined below).
- **Single-file front end.** Plain `index.html` + `styles.css` + `app.js`. No bundler, no Tailwind, no React. Use modern vanilla CSS (custom properties, `clamp()`, container queries, grid) and vanilla JS.
- **SSE-driven, live data.** The UI updates in real time from a server-sent-events stream. Layout must not jump or reflow jarringly when values update (reserve space, use tabular numerals, animate only transforms/opacity).
- **Korean + English content.** Copy is bilingual (KR UI labels, EN technical terms). Type system must handle Hangul and Latin gracefully at every size.
- **Keep the class names.** Reskin via CSS first; only touch `index.html`/`app.js` markup where structure genuinely needs it. Minimize churn.

## 1. Design thesis — what "doesn't look AI" actually means

Generic AI output has a recognizable signature. We are deliberately engineering *against* each tell.

| The AI tell (avoid) | What we do instead |
|---|---|
| Purple→blue 135° gradient on everything (buttons, logos, headings) | **One** restrained accent, used as ink — not as a gradient wash. Gradients, if any, are <8% opacity atmosphere, never on text or primary buttons. |
| Glassmorphism + heavy glow + neon shadows | Flat, confident surfaces. Depth comes from a **disciplined 1px border + one honest shadow**, not from blur and glow. |
| Everything equally rounded, equally spaced, equally weighted | **Deliberate hierarchy.** A few elements are loud; most are quiet. Intentional asymmetry. |
| Emoji as iconography (⚙️ 🚀 ✨ 📁) | A **cohesive icon language** — inline SVG, single stroke weight, one visual family. Emoji are removed from chrome. |
| Centered hero, three equal feature cards, generic spacing | A **real editorial grid** with an opinionated layout, an off-center focal point, and content-driven rhythm. |
| `box-shadow: 0 4px 6px rgba(0,0,0,.1)` defaults | A bespoke, tuned elevation scale (below). Shadows match the light source and the dark canvas. |
| Inter/Roboto at one weight everywhere | A **typographic system**: a display face with character + a precise UI/mono pairing, multiple weights used purposefully. |

**The north star:** the dashboard of a serious developer tool — think the craft level of Linear, Vercel, Stripe, Raycast, Datadog — but with *its own* identity, not a clone of any of them. A reviewer should think "a designer made this," never "this came out of a generator."

## 2. Brand identity — give auto-dr a face

A distinctive product needs a point of view, not just a palette.

- **Concept:** "an autonomous review *instrument*" — a precision measuring device for document/code quality. Lean into instrument-panel / control-room metaphors: gauges, readouts, signal, telemetry. This justifies dense data, monospaced numerics, and a calm, technical mood.
- **Logomark:** replace the ⚙️ emoji with a **custom inline-SVG mark** — a minimal monogram or an abstract "review loop" glyph (a rotating/recursive arrow, or an `a/dr` ligature). Must render crisp at 20–40px, work in a single color, and have a recognizable silhouette. Provide light-on-dark and monochrome variants.
- **Wordmark:** "auto-dr" set in the display face, tight tracking, lowercase, with a deliberate detail (e.g. the hyphen styled as a small accent tick, or `dr` in the accent ink). No gradient text.
- **Voice:** terse, expert, Korean-first labels (`자율 리뷰·개선 루프`). Numbers and statuses are the heroes.

## 3. Foundations — design tokens

Define everything as CSS custom properties on `:root` so the system is coherent and themeable. Values below are the *intended targets*; tune during implementation.

### 3.1 Color

Build a **perceptually even** dark palette (think OKLCH steps, even if authored as hex). Avoid muddy pure-gray; use a slightly cool, desaturated tint so it feels designed, not default-Bootstrap.

- **Canvas / surface ramp:** 5 steps from near-black canvas → elevated panel. Narrow range; surfaces are distinguished by *border + 1px*, not by big fills.
- **Ink ramp:** primary text (high contrast), secondary, muted, faint. Meet **WCAG AA (≥4.5:1 body, ≥3:1 large)** on every surface.
- **One accent.** Pick a single, ownable hue (NOT the default blurple #6d8bff→#9b7bff). Options to explore: a precise signal-green, an electric amber, a cyan-teal, or a refined cobalt — chosen so it reads "engineering instrument," not "generic SaaS." Define accent, accent-hover, accent-pressed, and a low-alpha accent-wash for selected states.
- **Semantic set:** success / warning / danger / info — each with a text tint, a border tint, and a low-alpha fill tint, so badges and alerts are consistent and never garish.
- **Data-viz scale:** a separate, categorical palette for charts/score dimensions that is colorblind-safe and harmonizes with the accent.

### 3.2 Typography

A real pairing, all self-hosted or system — no remote fonts.

- **Display / brand:** a face with personality for the logo + large numeric readouts (a tight grotesque or a distinctive variable sans). Self-host one weight range if used.
- **UI text:** a clean, dense humanist/grotesk sans with excellent Hangul support. Recommended stack (system-first, zero-download):
  `ui-sans-serif, -apple-system, "Segoe UI", "Pretendard", "Apple SD Gothic Neo", "Malgun Gothic", system-ui, sans-serif`.
- **Mono / numerics:** a monospace for telemetry, IDs, scores, code, diffs (`ui-monospace, "Cascadia Code", "JetBrains Mono", Consolas, monospace`). Use `font-variant-numeric: tabular-nums` everywhere numbers update live.
- **Type scale:** a modular scale (≈1.2–1.25 ratio) with **fluid sizing via `clamp()`** so headings scale with the viewport. Define `--step--1 … --step-6`.
- **Rules:** generous line-height for prose (1.6–1.7), tight for display (1.05–1.2). Set `letter-spacing` negative on large display, ~0 on body. Cap measure at ~70ch for any long-form text (review reports).

### 3.3 Spacing, radius, elevation, motion

- **Spacing:** an 8px-based scale exposed as tokens (`--space-1:4px` … `--space-12:96px`). Use it consistently; no magic numbers.
- **Radius:** a *restrained* scale — sharp enough to read as professional, not bubbly. e.g. `--r-xs:3px / --r:6px / --r-lg:10px`, plus a true circle only for status dots/avatars. (Calibrate to taste; the prior "too round" feedback means stay on the crisp end.)
- **Borders:** hairline `1px` borders are a primary design element. Define `--border` and `--border-strong`.
- **Elevation:** a 3-tier shadow scale tuned for a dark canvas (low spread, soft, slightly downward; never neon-colored). `--shadow-1` (resting card), `--shadow-2` (popover/toast), `--shadow-3` (modal).
- **Motion:** define `--ease-out`, `--ease-spring`, and durations (`--dur-fast:120ms`, `--dur:200ms`, `--dur-slow:380ms`). Animate **transform/opacity only**. Honor `@media (prefers-reduced-motion: reduce)` — disable non-essential motion.

## 4. Layout & responsive system (the core ask)

The site must be **genuinely responsive** — not a desktop layout that merely shrinks. Mobile, tablet, and wide-desktop each get a considered treatment.

### 4.1 Strategy

- **Fluid-first.** Use `clamp()` for type and spacing, `minmax()`/`auto-fit` grids, and **container queries** (`@container`) for components that should adapt to *their slot*, not just the viewport. Media queries only for top-level layout shifts.
- **Breakpoints (content-driven, not device-driven):**
  - `≤640px` **mobile** — single column; sidebar collapses to a top sheet / bottom-nav or a slide-over drawer triggered by a header button; the "new review" form becomes a full-width collapsible or a modal; cards stack; tables become stacked key/value rows or horizontally scroll with a sticky first column.
  - `641–1024px` **tablet** — two-column where it helps; sidebar may become a narrower rail or an overlay; metric cards in a 2-up grid.
  - `1025–1440px` **desktop** — the canonical 3-zone layout (sidebar · main · detail rhythm).
  - `≥1441px` **wide** — cap content max-width or introduce a comfortable multi-column reading/telemetry layout; don't let lines run edge-to-edge.
- **The shell:** header (sticky, condensed on scroll) + sidebar (run list + new-review) + main detail pane. On mobile the sidebar must be reachable and dismissible without trapping focus.

### 4.2 Responsive component rules

- **Sidebar:** desktop fixed rail (~320px); tablet collapsible; mobile off-canvas drawer with overlay + focus trap + `Esc` to close + body-scroll-lock.
- **Metric cards:** `repeat(auto-fit, minmax(150px, 1fr))` so they reflow naturally; never fewer than comfortable padding at any width.
- **Tables (iterations, findings):** on narrow screens, switch to a card/stacked layout via container query, OR an overflow-x scroll region with a pinned first column and a subtle fade affordance at the scroll edge.
- **Charts/score bars:** SVG/canvas must resize to container width; redraw on `ResizeObserver`. Keep a min-height so SSE updates don't cause layout shift.
- **Touch targets:** ≥44×44px on mobile; increase hit areas for icon buttons.
- **Forms:** inputs go full-width and 16px+ font on mobile (prevents iOS zoom); selects use a custom caret that scales.

### 4.3 Anti-shift discipline

Because data streams in: reserve space for values (min-width on numeric cells), use `tabular-nums`, skeleton/placeholder states while loading, and `content-visibility`/`contain` where long lists render. No spinner-induced reflow.

## 5. Components — the bar for "obviously well-made"

Each component gets craft details that signal a designer's hand.

- **Header:** sticky, translucent with a *subtle* backdrop blur (kept tasteful), a real SVG logomark, a live connection indicator (a calm pulsing dot, semantic colored), and quiet secondary actions. Condenses height on scroll.
- **Buttons:** a clear hierarchy — primary (solid accent, *no* gradient, crisp focus ring), secondary (1px border, surface fill), ghost (text + hover wash), and destructive. Consistent height tokens, optical padding, `:active` press feedback, visible `:focus-visible` ring (2px, offset).
- **Inputs / selects / dropzone:** unified field styling, custom select caret, drag-over state with a precise (not neon) highlight, clear filled/empty/error/disabled states, helper + error text slots.
- **Run list items:** information-dense rows — title (truncated with ellipsis), status chip, score, timestamp — with a refined selected state (inset accent bar, not a heavy fill). Hover reveals secondary affordances.
- **Status chips & severity badges:** small, sharp, semantic; consistent across the app; never rely on color alone (pair with icon/label) for accessibility.
- **Cards / sections:** flat surface + hairline border + one resting shadow; section headers as small uppercase eyebrow labels with letter-spacing; an optional top hairline accent on hero cards.
- **Findings ledger / progress bars:** thin, precise bars (2px radius), smooth width transitions, clear open/done states; struck-through done items at reduced opacity.
- **Toasts:** bottom-right stack, slide+fade in via transform, semantic left border, auto-dismiss with a subtle progress hairline, manual close, screen-reader `role="status"`/`aria-live`.
- **Empty/loading/error states:** designed, not default — a small illustrative SVG, a one-line explanation, and a clear next action. These states are where "well-made" is won or lost.
- **Data viz:** custom score chart with tasteful axis/gridlines, animated-in on load, tooltips, and the data-viz palette from §3.1.

## 6. Motion & micro-interaction

Tasteful motion is the difference between "template" and "crafted."

- Entrance: lists and cards fade+rise 6–10px on first paint, **staggered** slightly. One-time, not on every SSE tick.
- State changes: animate score/bar width with `--ease-out`; color transitions on hover/active at `--dur-fast`.
- Feedback: button press scale (~0.98), drop-zone highlight, copy-to-clipboard confirmation.
- Live data: when a value changes, a brief, subtle highlight (background flash → fade), never a layout jump.
- Respect `prefers-reduced-motion`: replace movement with instant opacity changes.

## 7. Accessibility & quality bar (non-optional)

"Obviously well-made" includes correctness, not just looks.

- **Contrast:** AA minimum on all text/surface combos; verify the accent on its backgrounds.
- **Keyboard:** full keyboard operability; logical tab order; visible `:focus-visible` rings; drawer/modal focus trapping + restore.
- **Semantics:** correct landmarks (`header`/`nav`/`main`/`aside`), `aria-live` for SSE updates and toasts, labelled form controls, `aria-current` on the active run.
- **Targets & input:** 44px touch targets, no hover-only affordances on touch, no iOS zoom traps.
- **Performance:** no layout thrash, `ResizeObserver` instead of resize spam, `content-visibility` for long logs, GPU-friendly animations only.
- **Reduced motion / forced colors / dark-mode native:** all handled.

## 8. Distinctiveness checklist (the "독보적" test)

Ship only if the design earns *yes* on these:

- [ ] Remove **every** chrome emoji; replace with a coherent inline-SVG icon set + a real logomark.
- [ ] Kill the default blurple gradient; commit to one ownable accent used as ink.
- [ ] Replace generic equal-card layouts with an opinionated, asymmetric, editorial grid.
- [ ] At least one signature detail that's *ours* (e.g. the instrument/telemetry numeric readouts, a distinctive run-list row, a custom score gauge).
- [ ] Typographic system with real hierarchy (display + UI + mono, multiple weights), fluid via `clamp()`.
- [ ] Tuned, bespoke elevation + a restrained crisp radius scale (not bubbly).
- [ ] Three genuinely distinct responsive treatments (mobile drawer, tablet rail, desktop/wide) — verified on real widths.
- [ ] Designed empty/loading/error states.
- [ ] AA contrast + full keyboard + reduced-motion verified.
- [ ] Nothing on screen looks like a default framework component.

## 9. Implementation plan (for the follow-up work session)

Order of operations to keep edits efficient and low-token:

1. **Tokens.** Rewrite `:root` in `styles.css` — color ramps, type scale (`clamp()`), spacing, radius, elevation, motion. This alone shifts 60% of the feel.
2. **Type & fonts.** Decide system-stack vs. self-hosted display face; add `@font-face` if hosting; wire the scale.
3. **Identity.** Add the inline-SVG logomark + icon set to `index.html`; remove chrome emoji.
4. **Shell & responsive skeleton.** Header condense-on-scroll, sidebar→drawer behavior (needs small `app.js` additions: drawer toggle, focus trap, `Esc`, scroll-lock), container/grid breakpoints.
5. **Components.** Reskin buttons, inputs, run list, cards, chips, ledger, toasts, tables (with mobile stacked variant).
6. **Motion & states.** Entrance stagger, live-value highlight, empty/loading/error designs, reduced-motion guard.
7. **QA pass.** Contrast, keyboard, touch, real-width responsive check, no-layout-shift on SSE.

> **Scope note for the editor:** prefer CSS-only changes; touch `app.js`/`index.html` only for (a) the SVG logomark/icons, (b) the mobile drawer behavior, and (c) `ResizeObserver`/skeleton hooks. Keep existing class names so the diff stays small.
