## 발견된 문제점

### 1. 🔴 비용 상한 초과가 "정상 종료"가 아니라 "크래시(error)"로 처리됨 — **사용자가 본 "끊김"의 정체**
- 증거(log): `반복 #3 실패: claude 오류 응답: error_max_budget_usd` → `재시도 #3...` → `status: error`, `message: 반복 #3 재시도 실패: ... error_max_budget_usd`
- 원인: 회차당 예산 = `min(없음, 총상한 0.5 − 누적 0.4416) = $0.058`. opus 호출에 `--max-budget-usd 0.058`이 걸려 **즉시 거부**. 러너가 이 거부를 *일시적 실패*로 보고 재시도 → 같은 예산으로 또 실패 → 런을 `error`로 마감.
- 영향: 베스트(85점)는 보존됐지만 런이 **에러로 표시**되어 마치 프로그램이 깨진 것처럼 보임. 실제론 돈이 떨어진 것.

### 2. 🔴 "예산 벽"으로의 자동 승급 — 자멸적 동작
- 증거(log): `2회 시도 소진 → 모델 승급 sonnet → opus 로 다음 회차 재시도`
- 원인: `maxAttemptsPerModel=2` 소진 시 더 강한 모델로 승급하는데, **남은 예산이 $0.058뿐**인 상태에서 가장 비싼 opus로 올라가 호출 → 100% 실패. 승급이 곧 죽음.

### 3. 🟠 후속 패스가 "말로만 해결" — edit 미생성 (수렴 느림)
- 증거(log): `부분 수정 1건 적용` 직후 `해결 주장 8건은 실제 수정이 확인되지 않아 보류(열린 상태 유지)`
- 의미: 2회차에 모델이 9개를 고쳤다고 주장했지만 **실제 edit은 1개만** 냈음. 정합성 가드가 나머지 8건을 열린 채로 유지(✅ 가드 정상 작동). 하지만 회차당 실효 수정이 1건뿐이라 **수렴이 매우 느림**.
- 부수효과: 열린 항목이 9 → 10으로 오히려 증가(해결 1 < 새발견 2).

### 4. 🟠 자가 채점 점수 인플레 — 점수↑인데 미해결↑
- 증거: 점수 72 → 85 로 13점 상승했는데, 같은 구간에 열린 지적은 9 → 10건으로 **증가**.
- 의미: 개선한 모델이 자기 결과를 채점하니 실제 미해결이 늘어도 점수가 오름. 정지·채택 판단의 신뢰도 저하.

### 5. 🟡 낮은 기본 상한이 조기 종료를 강제
- $0.50 상한에 2회차 만에 $0.44 소진 → 3회차 여력 없음. `maxIterations=3`도 빡빡. 수렴 전에 멈출 수밖에 없는 구조.

## 이번 세션에서 적용한 수정

| # | 문제 | 조치 | 상태 |
| --- | --- | --- | --- |
| 1 | 예산초과가 error로 크래시 | `isBudgetError()` 감지 → **`stopped_cost`로 graceful 종료**(재시도 안 함, 베스트 보존) | ✅ |
| 2 | 예산 벽으로 승급 자멸 | 호출 전 **예산 바닥 가드**(`MIN_CALL_BUDGET_USD=$0.03` 미만이면 승급/호출 대신 graceful 종료) | ✅ |
| 4 | 자가 채점 인플레 | **독립 채점기**(별도 모델이 최종본 채점) 도입 — 채택·정지 판단에 사용 | ✅ |
| 5 | 낮은 상한 조기 종료 | 기본 `maxTotalCostUsd $0.5→$2`, `maxIterations 5→8`, 세션 상한 $5→$10 | ✅ |
| 3(부분) | 후속 수정 빈약 | **마무리 정밀검증 패스**(완료 전 강한 모델로 전체 1회 점검) + **후속 패스 깊이 옵션화** | ✅ |

> 참고: 정합성 가드(해결 주장 보류)는 이 런에서 **이미 정상 작동**해 8건의 가짜 해결을 막았습니다(문제가 아니라 의도된 안전장치 동작).

## 권장 사항 진행 현황

- **R1. 후속 패스 edit 강제력 강화** — ✅ **적용**: 2회차+ 해결은 `findingIds` 태그로 뒷받침된 것만 인정(태그 없으면 적용 edit 수만큼만). "말로만 해결" 차단. → 문제 #3 근본 대응.
- **R2. 새 발견 억제(후반)** — ✅ **적용**: 후반 신규 지적 회차당 최대 3개(중요도 우선) + 프롬프트 "열린 항목 해결 최우선". → 대장 폭증 완화.
- **R3. 승급 정책에 예산 인지** — ✅ **적용**: 직전 회차 비용 × 모델 단가비로 상위 모델 1콜 추정비를 내고, 잔여 예산이 부족하면 **승급하지 않음**(예산 벽으로의 자멸적 승급 차단). `estimateModelCost`.
- **R4. docx/pptx 왕복 검증** — ✅ **적용**: 오피스 재생성 직후 다시 추출해 본문 손실을 점검(런타임 경고) + `test/export.roundtrip.test.ts`(docx·pptx 구조 보존 테스트).

## 결론
이 런이 "끊긴" 진짜 이유는 **토큰 한계가 아니라 $0.50 비용 상한 + 예산 벽으로의 opus 승급이 `error`로 처리된 것**이었습니다. 이번 수정으로 (1) 같은 상황이 `error`가 아니라 `stopped_cost`로 깔끔히 끝나고, (2) 애초에 돈이 부족하면 비싼 호출을 시도하지 않으며, (3) 상한을 현실적으로 올리고, (4) 독립 채점 + 마무리 패스로 품질·정지판단을 보강했습니다.
